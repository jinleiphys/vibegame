# vibe科研操作系统 · 项目规划

## 1. 项目定位

`vibe科研操作系统` 是一个单页网页游戏原型。

它的包装层是一个假的 AI 科研 IDE / agent 工作台；它的玩法内核是一个轻策略管理游戏。玩家不扮演某个具体学科的研究者，而是扮演“广义科研工作者 / PI / 独立研究者”，在有限资源下调度 AI、推进项目、抵御幻觉、冲刺投稿和基金。

这不是某个学科的模拟器，而是对现代科研工作流的抽象化、讽刺化和游戏化。

## 2. 设计支柱

### 支柱 A：假操作系统的代入感

- 界面必须像一个真的科研 agent 面板
- 终端、通知、资源条、项目板、系统警报都要服务于这层幻想
- 玩家应感觉自己在“运营一条科研生产线”

### 支柱 B：每一步都要做痛苦取舍

- 资源不能只是数字展示
- 任何推进都必须有代价：花钱、烧 tokens、消耗 judgment、掉 mental
- “快”和“稳”之间必须有持续张力

### 支柱 C：短反馈要强

- 点击一个动作就要立刻看到数值变化、文本后果、项目推进
- 反馈不能只在结局出现
- 每周循环必须能在几十秒内完成一次并产生爽感

### 支柱 D：重复可玩

- 每周任务
- 随机事件
- 多学科项目
- 不同资源状态触发不同局面
- 投稿 / 基金 / 审稿结果带来不同分支

### 支柱 E：真相要靠踩坑学出来

- 模型强弱不能在面板上一眼读完
- 平台稳定性要靠实测口碑慢慢积累
- 学生不是固定加成卡，而是会带着具体坏毛病进入系统
- 玩家应通过失败和试错学会“好模型很贵，坏平台更贵，人也不会自动稳定”

## 3. 玩家幻想

玩家的核心幻想不是“我会写 prompt”，而是：

> 我在一个极度高压的现代科研系统里，用 AI 放大执行力，但真正决定我成败的是我如何分配判断力。

## 4. 目标用户体验

玩家在 5 到 10 分钟的一局中，应当持续体验到以下情绪：

- 野心：我这一周能不能把项目冲过去
- 紧张：资源已经不够了
- 幽默：系统文本对科研现实有黑色幽默
- 成就：我做出了一次聪明取舍
- 后悔：我刚才那步太贪了
- 再来一周：我再试一轮应该能更优

## 5. MVP 核心循环

每周：

1. 看资源状态
2. 看当前主线项目
3. 看本周任务
4. 看限时事件
5. 先决定本周启用哪个模型
6. 再决定是否购买 tokens / 算力
7. 在 3 次行动内做取舍
8. 选择是否投稿 / 申请基金 / 安抚团队 / 防止爆雷
7. 进入下一周并刷新内容

这是 MVP 必须成立的核心 loop。

## 6. MVP 系统

### 6.1 资源系统

- `Cash`
- `Tokens`
- `Compute`
- `Judgment`
- `Reputation`
- `Mental HP`
- `Hallucination Risk`
- `Actions Left`
- `Team Stress`
- `Contract Clock`

### 6.2 项目系统

每个项目包含：

- 学科标签
- 标题
- 描述
- `Progress`
- `Novelty`
- `Correctness`
- `Polish`

### 6.3 模型系统

不是所有 LLM 都一样。

每个模型至少包含：

- token 成本倍率
- 进度提升倍率
- 正确性倍率
- 风险倍率
- 吞吐速度
- 队列阻塞风险

关键讽刺点：

- 强模型稀少且昂贵
- 平台说自己有最强模型，不代表真的给你足够 GPU
- 模型买对了，基础设施没配够，照样只是在更贵地等待
- 这些真相不会在 UI 上直接展示，而是伪装成供应商文案、组内口碑和踩坑经验

### 6.4 行动系统

首版动作：

- 购买 tokens
- 购买算力
- 文献扫描
- 代码 / 实验原型
- 写论文初稿
- 人工核验
- 跑 benchmark
- 申请基金
- 内部审计
- 切片灌水
- 休息
- 安抚学生 / 组会管理
- 投稿当前项目

### 6.5 团队系统

首版会有轻量人物卡和一个抽象的 `Team Stress`。

它代表：

- 学生情绪
- 合作者疲劳
- 实验室管理摩擦
- 导师提供情绪价值和安抚所需的额外劳动

成员还会带 trait，例如：

- 啥也不会但会把超算额度一周烧空
- 写得像论文但证据明显没跟上
- 静默蒸发型拖延
- 状态波动极大的脆皮天才
- 结果不够漂亮就想先修结果的造假风险

团队层的黑色幽默要落在机制上，而不是只落在一句话上：

- 导师的情绪劳动是硬成本
- 坑货成员会制造真实资源损失
- 造假和偷懒要么被内审抓住，要么晚点以更贵的方式回来

设计原则：

- 黑色幽默可以有
- 刀口主要对准制度和工作流荒诞性
- 不把学生做成纯粹的无脑笑料

### 6.6 任务系统

每周给出 3 个短目标，例如：

- 把某项目正确性拉到 60 以上
- 将风险压到 20 以下
- 保持 mental 高于 50
- 攒到一定 tokens
- 完成一次投稿

任务完成后立即发奖励。

### 6.7 随机事件系统

每周 1 个限时事件，玩家做 1 次选择。

事件类型：

- 论文事件
- 基金事件
- 合作者事件
- 舆论 / 平台 / 热点事件
- 基础设施 / GPU 队列事件
- 学生 / 团队情绪事件
- 作者署名政治
- 引用绑架 / citation cartel 味道
- 掠夺性会议 / 期刊
- 论文中介 / paper mill
- 开放数据 / 可重复性审查
- AI 审稿和新型灰色骚操作

### 6.8 结果系统

- 投稿可能：接收 / 大修 / 拒稿
- 基金可能：中 / 不中
- 资源可能触发：burnout、高风险警报、现金告急

## 7. 内容范围

### 学科池

- Climate Science
- Biology
- Digital Humanities
- AI Research
- Materials Science
- Social Science

### 事件池

MVP 目标：先做 8 到 12 个事件。

### 任务池

MVP 目标：先做 8 个任务模板。

## 8. UI 结构

单页布局：

- 顶部：产品标题、周数、行动点、模式
- Hero：产品说明 + 假终端
- 资源面板：核心资源 + 周任务
- 项目面板：项目切换和进度
- 动作面板：行动按钮 + 即时反馈 + 当前事件
- 事件流：系统日志 / 剧情后果

## 9. 表现策略

### 视觉

- 像操作系统，不像通用 SaaS
- 纸张感和终端感混搭
- 颜色体现风险和稀缺资源

### 文案

- 黑色幽默
- 短句
- 像系统提示，但带讽刺
- 能让懂科研的人一眼认出“这事真发生过”

### 动效

- 进度条更新要有过渡
- 关键状态要有脉冲或高亮
- 最近一次结果要有 banner 级反馈

## 10. 本轮执行计划

### 当前已实现

- 开局流派选择
- 本地存档 / 继续 run
- 多结局 run 结构
- 实验室成员卡与周结算
- 模型市场、tokens/compute 双资源
- 周任务、随机事件、即时反馈、可复制战报

### Phase 1

- 写好规划文档
- 把动作改成每周 3 次
- 增加模型市场
- 增加算力资源
- 增加周任务
- 增加随机事件
- 增加团队压力和安抚动作
- 增加更强的即时反馈

### Phase 2

- 扩大学科池和事件池
- 增加更多中期结果
- 增加风格化 UI 和音效占位

### Phase 3

- 本地验证
- 同步到 `~/Desktop/game`

## 11. 本轮不做

- 后端
- 登录系统
- 真正的 LLM API
- 长剧情分支
- 多页面架构
- 复杂存档系统

先把单页 MVP 做得可玩，再扩张。
